# Data Model: Authenticated Session And Current User

## Authenticated Identity

Represents a request principal whose identity has been proven by the configured identity provider.

**Fields**:

- `provider`: stable identifier for the identity provider configuration.
- `subject`: stable provider subject used to map to one local application user.
- `displayName`: optional human-readable name for current-user responses.
- `email`: optional email address for current-user responses and diagnostics.

**Validation rules**:

- `subject` is required for local user mapping.
- Missing or blank `subject` is treated as an authentication failure for current-user behavior.
- Provider authorization payloads are not stored or exposed as application authorization.

## Local User Identity

Application-owned representation of an authenticated person.

**Fields**:

- `id`: stable local user identifier generated by the application.
- `provider`: provider identifier from the authenticated identity.
- `subject`: provider subject from the authenticated identity.
- `displayName`: optional display value last observed from the provider.
- `email`: optional email value last observed from the provider.
- `createdAt`: timestamp when the local user identity was first created.
- `lastSeenAt`: timestamp when the authenticated identity was most recently resolved.

**Relationships**:

- One local user identity maps to exactly one `(provider, subject)` pair.
- One local user identity may have zero or one active application access record in this first slice.

**Validation rules**:

- `(provider, subject)` must be unique.
- Repeated requests with the same `(provider, subject)` must resolve the existing local user identity.
- Local user identity data must not expose raw provider tokens.

## Application Access Record

Application-owned authorization record that determines whether a local user can access protected application behavior.

**Fields**:

- `id`: stable access record identifier.
- `localUserId`: local user identity the record grants access to.
- `isActive`: whether the record currently grants application access.
- `createdAt`: timestamp when the record was created.
- `disabledAt`: optional timestamp when access stopped being active.

**Relationships**:

- Belongs to one local user identity.
- Determines the `hasApplicationAccess` state returned by current-user behavior.

**Validation rules**:

- Access decisions are based only on application-owned records.
- Missing, inactive, or disabled access records mean the user is authenticated without application access.
- Provider roles, groups, organizations, and provider-specific claims are ignored for product authorization.
- The public current-user contract exposes only whether application access exists; product-specific roles and access levels are deferred.
- Narrow initial access/bootstrap data may exist only to seed or verify application-owned access from configuration.
- Bootstrap behavior identifies one initial access provider/subject pair, is idempotent, creates no product-specific role model, and exposes no UI/API provisioning workflow.

## Current User

Request-time view used by backend/API behavior.

**Fields**:

- `isAuthenticated`: true when the API request has a valid authenticated identity.
- `localUserId`: stable local user identifier when authenticated and mapped.
- `displayName`: optional display name safe for clients.
- `email`: optional email safe for clients.
- `hasApplicationAccess`: true when an active application access record exists.

**State transitions**:

```text
Unauthenticated request
└── API authentication required -> 401 Unauthorized

Authenticated identity with stable subject
└── Local user missing -> create local user with no default access
    ├── Active application access exists -> authenticated with access
    └── No active application access -> authenticated without access

Authenticated identity missing stable subject
└── Cannot map local user -> 401 Unauthorized
```

## Deferred Server-Side Ticket Storage

Represents full server-side browser session ticket storage planned by the platform docs but deferred from this slice.

**Fields**:

- `sessionKey`: opaque session reference held by the browser.
- `ticket`: server-side authenticated session ticket.
- `expiresAt`: session expiration instant.

**Validation rules**:

- Full OIDC challenge/callback/logout mechanics and Redis-backed ticket storage are out of scope for this slice.
- This slice still verifies API authentication failure behavior: missing, expired, invalid, or unmappable authenticated context produces `401 Unauthorized` for API requests requiring authentication.
- API authentication and authorization failures must not redirect to browser login pages.
- Session ticket storage remains a Host/platform concern, not Identity application data.
